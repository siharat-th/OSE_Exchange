Efvitcp is a tcp library using Solarflare ef\+\_\+vi interface on linux, and also a tcp multiplexing framework for both C++ client and server program.

As ef\+\_\+vi is a low-\/level interface passing Ethernet frames between applications and the network, efvitcp implements full tcp/ip stack on its own in user space while also taking advantage of ef\+\_\+vi\textquotesingle{}s zero-\/copy buffer delivering feature.

On the user interface, efvitcp is a header-\/only C++ template for tcp client and server, using reactor model(like select/poll/epoll) and handling events for multiple connections in a single call.

It opens a fast path directly between the user and the NIC, providing extremely efficient data processing performance. For example, on frame arrives the tcp payload memory could be directly handed over to the user which is written by the NIC on a DMA buffer.\hypertarget{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md1}{}\doxysection{\texorpdfstring{Why not use Tcpdirect?}{Why not use Tcpdirect?}}\label{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md1}
While tcpdirect also provides a tcp implementation based on ef\+\_\+vi, it suffers from the maximum PIO number limitation the solarflare NIC provides\+: there can be no more than 12 instances on a host(onload is sharing the same PIO resource which makes it worse). And tcpdirect performs poorly in performance when onload is also in use, which I don\textquotesingle{}t know why. These problems make tcpdirect hard to use in practice, so an alternative approach is needed to make full use of solarflare NIC\textquotesingle{}s low-\/latency capacity for tcp applications.\hypertarget{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md2}{}\doxysection{\texorpdfstring{Features}{Features}}\label{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md2}

\begin{DoxyItemize}
\item Fast -\/ it makes good use of solarflare\textquotesingle{}s low-\/latency features including CTPIO, it\textquotesingle{}s even faster than tcpdirect especially for large segments.
\item Headers only and no third-\/party dependences(except for ef\+\_\+vi).
\item Reactor model with rich network and time events, giving finer control over a tcp connection.
\item Highly configurable in compile-\/time.
\item Non-\/blocking -\/ none of the apis blocks.
\item No thread created internally.
\item Supports user timers on each connection.
\item Provides the same interface as other pollnet tcp libs, such as Socket and Tcpdirect.
\end{DoxyItemize}\hypertarget{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md3}{}\doxysection{\texorpdfstring{Platform}{Platform}}\label{md__c_1_2_users_2sgaer_2source_2repos_2katanafinancial_2_occam_c_m_e4_2_occam_c_m_e4_2src_2exter7253da91f16cb30ff9cbb16d01f0338c_autotoc_md3}

 Note that {\ttfamily connect()} not returning an error doesn\textquotesingle{}t mean the connection is established\+: it just indicate that a Syn is sent out without error(as it\textquotesingle{}s non-\/blocking), we still need to wait for the 3-\/way handshake to complete. As efvitcp won\textquotesingle{}t create a thread internally, we need to {\ttfamily poll()} it in the user thread, and as {\ttfamily poll()} is also non-\/blocking, we need to call it {\bfseries{repetitively}}.

{\ttfamily poll()} is the core of muliplexing in the lib, all sorts of events will be triggered by this single call including whether or not the {\ttfamily connect()} is successful. How can user retrieve the events from {\ttfamily poll()}? A class with event handler functions(non-\/virtual) need to be defined and an instance be provided as the argument of {\ttfamily poll()}, and it\textquotesingle{}ll call those user defined callback functions when events occur.  {\ttfamily on\+Data()} callback returns the remaining size not consumed by the user, probably because only a part of application packet has been recevied and more data is needed to proceed. In this case user don\textquotesingle{}t need to buffer the partial data himself but just return the remaining size. When new data is avaialble, it will be appended to the old data and handed together to the user in the next {\ttfamily on\+Data()}, this allows for elegant data processing code just as above.

If user is done with sending and want to start or continue with the 4-\/way handshake closure, {\ttfamily send\+Fin()} can be called, this is analogous to {\ttfamily shutdown(fd, SHUT\+\_\+\+WR)}.  and the connection is closed. In addition, the {\ttfamily on\+Connection\+Timeout(Tcp\+Conn\& conn)} event can also occur on an established connection, indicating data retransmission time has exceeded some limit and the connection is automatically closed.

Note that after a connection is closed(either by active {\ttfamily close()} or on events {\ttfamily on\+Connection\+Closed}/{\ttfamily on\+Connection\+Reset}/{\ttfamily on\+Connection\+Timeout}) the connection reference should not be used because it could be reused for a new connection.

Lastly, efvitcp allows for user timers on a connection basis, and each connection can have multiple user timers concurrently. This is done by {\ttfamily void set\+User\+Timer(uint32\+\_\+t timer\+\_\+id, uint32\+\_\+t duration)} function on {\ttfamily Tcp\+Conn}, where {\ttfamily timer\+\_\+id} is the timer type starting from 0, with different timers having different {\ttfamily timer\+\_\+id}. {\ttfamily duration} is the timeout in milliseconds, currently the maximum possible duration is 65000(65 seconds). Setting a duration of 0 will delete a specific timer. When the connection is closed all users timers are deleted as well. When user timer expires below event will trigger\+:

\begin{DoxyItemize}
\item {\ttfamily uint32\+\_\+t Conn\+Send\+Buf\+Cnt}\+: The number of DMA send buffers in one connection. Send buffer is used to hold unacked data in case it need to be resent, if the buffer is full no more data can be sendable.
\item {\ttfamily bool Send\+Buf1K}\+: Whether to use 1024 bytes for a DMA send buffer. If set to false, 2048 bytes is used. ef\+\_\+vi requires that a DMA buffer be in a 4096 aligned block, and as typical MTU is 1500 bytes, using 2048 sized buffer will have 25\% memeory wasted. So setting {\ttfamily Send\+Buf1K} to true will maximize memory unilization but reducing the max SMSS to 960 bytes.
\item {\ttfamily uint32\+\_\+t Conn\+Recv\+Buf\+Size}\+: Receving buffer size in bytes for each connection. It determines the maximum receive window advertised to the remote peer, and also used to hold unconsumed data(by remaining size returned by on\+Data) and out of sequence data received.
\item {\ttfamily uint32\+\_\+t Max\+Conn\+Cnt}\+: Max number of connections supported. This is used only by Tcp\+Server.
\item {\ttfamily uint32\+\_\+t Max\+Time\+Wait\+Conn\+Cnt}\+: Max number of TIME\+\_\+\+WAIT connections supported. Note that TIME\+\_\+\+WAIT connection is not included in {\ttfamily Max\+Conn\+Cnt}. A TIME\+\_\+\+WAIT connection uses much less memory than a normal connection. Also note that efvitcp doesn\textquotesingle{}t distinguish between active and passive closing, and all connections completing the 4-\/way handshake will enter into TIME\+\_\+\+WAIT.
\item {\ttfamily uint32\+\_\+t Recv\+Buf\+Cnt}\+: The number of DMA receive buffers(one is 2048 bytes) shared by all connections. These DMA buffers are used to transfer network frames received by the NIC to the application, once efvitcp got a filled buffer it can be reused again immediately, so this config don\textquotesingle{}t need to be large, typically 512 is a good default value, the maxmium number ef\+\_\+vi can support is 4095.
\item {\ttfamily uint32\+\_\+t Syn\+Retries}\+: Max number of SYN resents for an unestablished connection. When breached, on\+Connection\+Timeout will be triggered.
\item {\ttfamily uint32\+\_\+t Tcp\+Retries}\+: Max number of data segment resents for an established connection. When breached, on\+Connection\+Timeout will be triggered.
\item {\ttfamily uint32\+\_\+t Delayed\+Ack\+MS}\+: Timeout of delayed ack. Setting to 0 will disable delayed ack.
\item {\ttfamily uint32\+\_\+t Min\+Rto\+MS}\+: Mininum retransmission timeout.
\item {\ttfamily uint32\+\_\+t Max\+Rto\+MS}\+: Maximum retransmission timeout.
\item {\ttfamily bool Window\+Scale\+Option}\+: Whether or not to enable tcp window scale option. This option is useful if window size could be larger than 65535 in either peer.
\item {\ttfamily bool Timestamp\+Option}\+: Whether or not to enable tcp timestamp option.
